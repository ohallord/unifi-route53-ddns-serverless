# Contributing to unifi-route53-ddns-serverless

Welcome! We appreciate your interest in contributing to the **unifi-route53-ddns-serverless** project. Your feedback, bug reports, and feature suggestions are invaluable.

This document outlines the guidelines for contributing to this project.

## How Can I Contribute?

### ðŸ’¡ Reporting Bugs

If you find a bug, please help us by [opening an issue](https://github.com/ohallord/unifi-route53-ddns-serverless/issues/new?assignees=&labels=bug&projects=&template=bug_report.md&title=). Before submitting a bug report, please check the existing issues to see if your problem has already been reported.

When reporting a bug, please include:
*   A clear and concise description of the bug.
*   Steps to reproduce the behavior.
*   Expected behavior.
*   Screenshots (if applicable).
*   Your environment details (e.g., Serverless Framework version, AWS Region, UniFi firmware version).

Do you have an idea for a new feature or an improvement to an existing one? We'd love to hear it! Please [open an issue](https://github.com/ohallord/unifi-route53-ddns-serverless/issues/new?assignees=&labels=enhancement&projects=&template=feature_request.md&title=) to suggest an enhancement.

When suggesting an enhancement, please describe:
*   The problem you're trying to solve.
*   Your proposed solution.
*   Any alternatives you've considered.

### ðŸ’» Code Contributions (Pull Requests)

We welcome code contributions! If you'd like to contribute code, please follow these guidelines:

1.  **Discuss First**: For major changes or new features, please [open an issue](https://github.com/ohallord/unifi-route53-ddns-serverless/issues/new/choose) first to discuss your idea with the maintainer. This helps avoid duplicate work and ensures your contribution aligns with the project's goals.
2.  **Fork the Repository**: Fork this repository to your own GitHub account.
3.  **Create a New Branch**: Create a new branch for your changes (e.g., `feature/your-feature-name` or `fix/bug-description`).
4.  **Make Your Changes**: Implement your changes, ensuring your code adheres to the existing style and conventions.
5.  **Test Your Changes**: If applicable, ensure your changes are tested.
6.  **Commit Your Changes**: Write clear, concise commit messages.
7.  **Submit a Pull Request**: Push your branch to your fork and open a Pull Request against the `main` branch of this repository.

**Please note**: All Pull Requests will be reviewed by the maintainer. While we appreciate all contributions, the maintainer reserves the right to accept, reject, or request modifications to any Pull Request based on project scope, quality, and maintainability.
